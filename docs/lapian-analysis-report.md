# Lapian Notes 项目分析与产品机会报告

> 核查日期：2026-07-09

## 1. 核心结论

Lapian Notes 选中了一个真实且有扩展空间的创作场景：把长视频转化为与时间码对齐、可人工修改的结构化拉片笔记。它最有价值的部分是完整工作流设计：本地处理影片、生成 AI 分析包、导回结构结果、人工精修、导出作品。这个闭环比单次“让 AI 分析电影”更接近可长期使用的工具。

如果开发类似产品，推荐沿“专业桌面工作台 + 本地优先 + AI 可选”的方向推进。首发版应完整复刻参考项目的价值链，同时把镜头自动切分、可靠项目存储、任务恢复、版本历史、搜索、专业导出和更成熟的视觉体系作为差异点。

推荐产品定位：**影视拆解与创作学习工作台**。首批用户聚焦导演、编剧、剪辑师、影视学生、教师和影评创作者。

## 2. 核查范围

本报告核查了：

- 仓库 README、MIT 许可证和 `package.json`。
- 当前核心类型、AI 导入、媒体处理、字幕、项目存储、导出、时间线和检查器实现。
- 首发以来的近期提交记录。
- 公开 Issue 与 PR 状态。
- Vite 8 和 Electron 的官方运行与安全要求。

事实边界：仓库在核查时刚完成首次发布并快速迭代；公开 Issue/PR 为空不能证明成熟稳定。仓库搜索索引未返回代码搜索结果，因此关键实现通过直接读取源文件和提交差异核对。

## 3. 项目做了什么

### 3.1 用户闭环

1. 用户导入影片。
2. 系统本地转码、抽帧、读取或寻找字幕。
3. 系统把截图、字幕、剧情资料和指令打成 ZIP。
4. 用户把 ZIP 交给 ChatGPT 等任意 AI。
5. 用户导回 JSON。
6. 系统生成剧情泳道、结构树和观众情绪曲线。
7. 用户逐段精修，并导出 Markdown、剧本或项目包。

### 3.2 已确认功能

| 能力 | 当前状态 |
|---|---|
| 固定间隔抽帧 | 已实现，默认按秒级采样 |
| 不兼容格式转码 | 已实现，本地 Vite 中间件调用 FFmpeg |
| 内嵌字幕 | 已实现，转码流程可提取文本字幕 |
| 网络字幕 | 已实现，按候选和片长做匹配与拒绝 |
| AI 分析包 | 已实现，模型无关、无 API Key 依赖 |
| AI JSON 导入 | 已实现，含预览与填空/追加/覆盖策略 |
| 剧情泳道 | 已实现，多故事线和复用卡片 |
| 结构树 | 已实现，按结构与故事线组织 |
| 情绪曲线 | 已实现，含强度、效价、类型和节奏角色 |
| 段落深拆 | 已实现，可生成单段分析包 |
| 人工编辑 | 已实现，字段较完整 |
| 项目库 | 已实现，IndexedDB 保存项目文本数据 |
| 备份迁移 | 已实现，自包含 ZIP |
| 导出 | Markdown 与剧本正文 |
| 一键启动 | Windows 和 macOS 脚本 |

## 4. 技术实现观察

### 4.1 技术栈

- React 19.2.6
- TypeScript 6.0.2
- Vite 8.0.12
- Vite 插件充当本地 Node 服务
- 浏览器 localStorage + IndexedDB
- FFmpeg/ffprobe 外部程序

产品依赖很少，便于理解和快速迭代。媒体工作负载和生产桌面应用的边界已经超出普通网页，后续继续使用 Vite dev server 承载本地服务会增加打包和维护压力。

### 4.2 代码规模信号

| 文件 | 约行数 | 含义 |
|---|---:|---|
| `src/App.tsx` | 1336 | 大量流程协调集中在一个组件 |
| `src/components/FrameTimeline.tsx` | 1039 | 多视图、时间线和交互密度高 |
| `src/components/InspectorPanel.tsx` | 633 | 编辑字段丰富，组件继续增长的概率高 |
| `src/lib/framePackage.ts` | 594 | 项目包、AI 包和导入导出责任较多 |
| `src/lib/markdown.ts` | 602 | 导出规则已形成独立复杂域 |
| `src/lib/aiImport.ts` | 523 | AI 结果兼容与校验是核心风险区 |

这些数字不自动等于代码质量问题，但说明后续开发应把“状态协调、领域命令、存储、视图渲染”分开，避免核心文件继续线性增长。

### 4.3 数据模型亮点

当前类型已经覆盖：

- Project、Frame、Subtitle、Segment、StoryLine。
- 段落结构功能、关键节拍、剧本块、创作意图、信息控制、节奏、技巧、观众体验和可复用方法。
- 情绪强度、效价、类型、节奏角色和关联段落。
- AI 置信度、人工/AI 来源、锁定等概念。

这说明项目的价值重点已经从“帧截图工具”进入“影视叙事知识建模”。

## 5. 做得好的地方

### 5.1 价值链完整

项目没有把 AI 输出停留在聊天文本。结果被映射为时间段、故事线、结构节点和情绪点，之后还能人工修改和导出。这为长期项目、教学和复盘提供了基础。

### 5.2 本地隐私与成本合理

原片和笔记默认留在本机。对长视频来说，本地抽帧和代理还减少上传等待与云存储成本。

### 5.3 模型无关

ZIP/JSON 往返让产品可以随模型能力与用户偏好变化。模型供应商故障或涨价时，核心产品仍然可用。

### 5.4 人机协同可解释

用户可以选择填空、追加、覆盖，段落还有置信度和来源信息。这个方向能减少 AI 一次性改坏人工笔记的风险。

### 5.5 首发后的修复优先级正确

提交记录显示作者优先处理：

- 新手无需预装 Node 的一键启动。
- macOS 启动与行尾兼容。
- 无字幕时仍可完成分析包。
- 错配字幕直接拒绝，防止污染 AI 判断。
- 项目自动保存到项目库。
- 片名不一致时允许用户确认导入共享结果。

这些问题都位于主闭环，修复顺序具有产品意识。

## 6. 主要短板

### 6.1 运行要求存在明确矛盾

README 写 Node.js 18+，项目使用 Vite 8。Vite 8 官方要求 Node.js 20.19+ 或 22.12+。一键启动脚本如果下载了低版本运行时，用户可能在安装阶段失败。

### 6.2 生产架构边界偏弱

转码和字幕搜索通过 Vite 开发服务器插件实现。README 已说明静态构建会降级。对于面向普通用户的完整产品，媒体处理应进入正式桌面主进程或独立 worker。

### 6.3 存储可靠性有限

localStorage、IndexedDB 和浏览器文件限制适合原型，长期项目会遇到配额、清理、迁移、备份、事务和跨浏览器差异。SQLite 与明确项目目录更适合可恢复的创作资产。

### 6.4 测试入口缺失

`package.json` 只有 dev、build、lint、preview，没有 test。时间映射、字幕编码、JSON 合并、ZIP 导入和数据库迁移都属于高回归风险模块。

### 6.5 抽帧粒度与电影语法不一致

固定 1 秒抽帧会漏掉快速剪辑，也会为长镜头生成大量重复帧。镜头边界检测应成为一级实体，固定间隔抽帧作为补充策略。

### 6.6 缓存身份可能碰撞

转码服务以文件名和大小生成短 ID。同名同大小的不同文件可能复用错误缓存。产品版应采用资产 UUID 与分块哈希。

### 6.7 网络字幕有稳定性和合规风险

实现依赖第三方页面结构、压缩格式和系统 `tar` 行为，也涉及字幕许可。正式版应优先支持内嵌字幕、旁挂字幕、本地 ASR，把网络字幕放到可选提供方。

### 6.8 UI 信息密度需要进一步治理

当前工作区功能丰富，CSS 使用偏传统的浅色企业工具风格。随着镜头、段落、AI、证据、统计继续增加，需要更清晰的层级、面板折叠、专注模式、命令面板和一致设计 Token。

## 7. 值得挖掘的产品机会

### 7.1 镜头级自动分析

自动检测镜头边界，并提取景别、镜头运动、构图、色彩、声音和对白。每个标签保留置信度与证据帧，用户可以批量校订。

价值：把抽帧升级为真正的镜头语法分析。

### 7.2 层级结构时间线

建立“幕/章节 → 段落/场景 → 镜头 → 帧/字幕”的层级。结构树、泳道、情绪曲线和播放器共享同一时间坐标。

价值：用户可以在全片结构与具体镜头之间快速往返。

### 7.3 个人影视知识库

提供全文检索、标签、人物、技巧、结构角色和情绪类型筛选。后续加入跨片语义搜索。

价值：每次拉片都会增加可复用资产，形成留存壁垒。

### 7.4 双片对比

同步播放两部作品，把同类段落、节奏曲线、平均镜头长度和景别分布对齐。

价值：适合学习同类型片、翻拍、广告竞品和导演风格。

### 7.5 可验证 AI

每条 AI 结论展示证据帧、字幕、时间码、置信度和模型版本。低置信度进入待确认队列。

价值：提升专业用户信任，也方便教学讨论。

### 7.6 专业交换格式

导出 PDF、CSV、FCPXML、EDL、SRT/VTT、联系表图片。

价值：分析结果能进入剪辑、论文、课程和编剧流程。

### 7.7 分析模板

为电影、短剧、广告、MV、动画、游戏过场提供不同字段、提示词和视图默认值。

价值：减少空白页压力，并扩大使用场景。

### 7.8 本地 ASR 与离线分析

提供 whisper.cpp 等可选本地转写适配。未来可以加入本地视觉模型，实现完全离线。

价值：覆盖未发布素材和敏感项目。

## 8. 建议的竞争策略

### 8.1 产品层

保持参考项目的低门槛闭环，同时提供更专业的镜头层、证据层和版本层。新用户通过四步向导完成第一次拉片，专业用户通过快捷键、命令面板和多轨时间线高效工作。

### 8.2 技术层

桌面应用使用 Electron，媒体任务进入 worker，项目数据进入 SQLite，项目资产使用可迁移目录。AI 适配器与导入 Schema 独立，手动分析包永久保留。

### 8.3 数据层

把所有结论绑定时间码和证据 ID。用户的人工修订、锁定字段和检查点属于最重要的长期数据。

### 8.4 商业层

基础本地版免费，Pro 提供 BYOK、版本历史、对比工作台、专业导出和批量项目；团队协作留到产品闭环与付费意愿得到验证后。

## 9. 推荐的最小发布范围

首个可发布版本必须有：

1. Windows/macOS 安装包。
2. 项目库与 SQLite 自动保存。
3. 媒体探测、代理、镜头检测、缩略图、波形。
4. 字幕导入、提取、编辑，可选本地 ASR 适配层。
5. 多轨时间线与播放器联动。
6. 泳道、结构树、情绪曲线、镜头网格。
7. 完整人工编辑。
8. 手动 AI 包与安全 JSON 导入。
9. 可选 BYOK。
10. 检查点、项目包、Markdown/PDF/CSV/字幕导出。
11. 单元、集成、E2E 与打包冒烟测试。

推荐延后：云同步、多人实时协作、移动端编辑、云端媒体托管。

## 10. 主要风险与缓解

| 风险 | 触发方式 | 缓解 |
|---|---|---|
| 媒体格式碎片化 | 特殊编码、可变帧率、多流 | ffprobe 探测、代理、合法媒体夹具矩阵 |
| 长任务失败 | 断电、取消、磁盘不足 | 持久化任务、原子产物、重试与清理 |
| AI 结构漂移 | 非法 JSON、错误时间和引用 | Schema、语义校验、diff、事务、撤销 |
| 数据损坏 | 迁移中断、并发编辑 | SQLite 事务、备份、检查点、迁移回滚 |
| UI 性能 | 1000 镜头、10000 字幕 | 虚拟化、Canvas、性能基准 |
| 密钥泄漏 | 日志、项目包、崩溃信息 | 系统钥匙串、脱敏、泄漏扫描 |
| 字幕合规 | 第三方抓取与再分发 | 内嵌/旁挂/ASR 优先，提供方按许可启用 |
| 范围失控 | 过早做云与协作 | 用 V1 Definition of Done 锁定边界 |

## 11. 建议开发节奏

- 第 1 周：桌面安全壳、设计系统、测试基线。
- 第 2–3 周：项目、SQLite、迁移、媒体探测和任务队列。
- 第 4–6 周：镜头、字幕、波形、虚拟化时间线。
- 第 7–9 周：分析领域、四种视图、编辑器、撤销重做。
- 第 10–11 周：AI 包、导入校验、BYOK。
- 第 12–14 周：版本、导出、打包、安全与性能硬化。

内部 MVP 可在 6–8 周形成；可靠 V1 的单人全职目标约 12–16 周。媒体管线完成后需要用实测重新估算。

## 12. 最终判断

这个方向值得做，原因有三点：用户工作流清晰、原始素材天然适合本地处理、结构化时间对齐数据能持续积累。产品成功的关键在于把媒体可靠性和人工编辑体验放在 AI 炫技之前。只要首版能让用户在没有 AI 的情况下完成一份专业拉片，再用 AI 显著加速，产品就拥有稳固的基础价值。

## 第一部分：第一性原理分析

### 1. 不可化简的任务是什么？

当一位导演、剪辑师、学生或影评人"分析一部电影"时，如果剥离所有软件、所有工具，只剩下纸和笔，他们在做什么？

**真实行为**（不是软件功能）：

1. **建立时间参照系**：把影片按时间轴切分成可定位的片段。在纸上，这意味着写下时间码或画面编号。这不可绕过——没有时间锚点，任何分析都是漂浮的。

2. **识别结构单元**：找到哪里开始、哪里转折、哪里结束。幕、场景、节拍——这些不是软件发明的概念，而是叙事认知的硬通货。人脑需要把连续流分解成命名单元才能操作。

3. **标注关键证据**：某个画面、某句台词、某个镜头运动——把它标出来，作为某个论点的支撑。分析的本质是"论点 + 证据 + 时间定位"。

4. **建立关系网络**：角色弧线、主题呼应、视觉母题重复——这些东西跨越时间，不能只看相邻帧。关系网是分析质量的分水岭，也是AI最容易失真的地方。

5. **产出可传播的判断**：论文、教学笔记、剪辑决策依据、创作方法论——分析的终点是让另一个人（或未来的自己）理解"为什么这样拍有效"。

**关键洞察**：当前软件把上述行为拆成功能（时间线、标签、笔记），但真正的认知单元是**时间锚定的证据-论点对**。换言之：用户在时间轴上打点，附着观察，建立联系，然后组织成叙事。软件存在的唯一理由，是让这个循环更快、更省记忆、更可复用。

### 2. 原子原语

以下是影视分析不可再分的单元。每个都需要解释为什么不可再分。

| 原语 | 定义 | 为什么不可化简 |
|---|---|---|
| **帧（Frame）** | 时间轴上的最小采样点 | 影视的基础物理单元。任何时间定位最终都要落到帧。即便用户不说"第 4853 帧"，说"2分41秒"，系统也必须能映射回去。不可再分，因为帧是视频存在的最基本形式。 |
| **镜头（Shot）** | 一次开机到停机 | 导演决策的最基本单元。景别、角度、运动、时长——所有这些变量都在镜头层面有意义，在帧层面无意义。不可再分，因为它是"一个完整创作意图"的最小容器。 |
| **时间码（Timecode）** | 时间的坐标 | 所有原语共享的坐标系统。没有它，镜头、字幕、标注、情绪全部失联。不可再分，因为它不是内容而是坐标——就像没有经纬度的地图毫无意义。 |
| **字幕/台词（Subtitle/Line）** | 带时间范围的口语内容 | 叙事内容的最小携带者。它有自己的时间跨度（可能跨多个镜头），不受镜头边界约束。不可再分，因为台词的时间逻辑和镜头的时间逻辑是两套独立体系。 |
| **段落/结构单元（Segment）** | 叙事功能上的一致性区域 | 幕、场景、节拍——它们覆盖多个镜头，定义的是"这一段在干什么"而非"这一段有几个镜头"。不可再分，因为叙事逻辑不从属于技术边界。一段戏可能在一个镜头内完成，也可能在30个镜头内完成。结构是独立的维度。 |
| **情绪标记（Emotion Point）** | 某个时点的情感状态 | 强度+效价+标签。它可以在帧层面、镜头内或跨场景。不可再分，因为情感分析的结果（"这里紧张"）与结构分析（"这是第二幕中段"）是正交信息。你不能用"这是第二幕"替代"这里观众应该害怕"。 |
| **证据链接（Evidence Link）** | 论点到证据的指向关系 | "这个镜头是荷兰角，表达了不安"——这是两个数据点加上一个关系。不可再分，因为分析的价值不在数据点在关系。没有链接，你只有一个镜头数据库和一个笔记本，它们不互通。 |
| **标签（Tag）** | 跨维度的分类标记 | 色彩、构图、象征、角色——标签跨越所有其他原语。一个标签"红色"可以贴在镜头、段落、情绪点上。不可再分，因为标签系统是与结构正交的检索维度。 |

**缺失的原语**（当前 spec 未明确定义的）：

- **角色（Character）**：角色进入/退出/在场是一个独立的时间维度。当前 spec 把角色信息藏在 segment 的 `fields_json` 里。这是一个**错误**——角色是独立实体，有入场/退场时间、弧线、与段落的交叉关系。标签不能替代它。
- **声音事件（Audio Event）**：音乐入场、音效冲击、静默——spec 提到波形但没有声音事件模型。这是**严重缺口**——对导演和剪辑师来说，声音设计至少占分析价值的一半。
- **视觉母题/重复模式（Visual Motif）**：跨镜头、跨场景的视觉重复模式。这不是标签能解决的——它需要一个"模式识别→命名→锚定多个证据"的工作流。

**总结**：当前 spec 的原语设计偏向文本叙事分析（镜头+段落+字幕+情绪），对声音和视觉特征建模不足。这不是 V1 功能缺失——这是数据模型的原语缺失。如果在 V1 的数据模型中不留位置，V2 加入时将是破坏性迁移。

### 3. 人/AI/自动化的边界

#### 只能人类做

| 任务 | 原因 |
|---|---|
| 判断"导演为什么用这个景别" | 需要对创作意图的推测，涉及语境、文化背景、导演个人风格——AI不会"理解"，只会统计相似模式 |
| 识别反讽和潜台词 | "这段对话表面在说A实际在说B"——需要理解人类交流的多层语义和戏剧冲突规则 |
| 建立跨作品的方法论语料库 | "这是我发现的第五个用负空间表达孤独的案例"——这是个人知识建构，AI只能辅助检索 |
| 决定"这个分析有教学/发表价值" | 价值判断不可外包——需要知道谁会在什么场景下使用这些分析 |
| 创作新的分析框架 | "我想用三幕剧+英雄之旅的混合框架分析这部非线性叙事"——框架创新是人的领域 |
| 锁定/定稿（Final Sign-off） | 任何专业工作流的最终批准权必须在人手中 |

#### AI 应该辅助

| 任务 | AI 的角色 | 人的角色 |
|---|---|---|
| 镜头边界检测 | 生成候选+置信度 | 确认、合并、拆分、调整敏感度 |
| 字幕转写/翻译 | 生成初稿 | 校对、修正专有名词和时间对齐 |
| 结构分段建议 | 基于模式识别给出候选分段 | 确认叙事逻辑、调整边界、命名 |
| 情绪曲线生成 | 基于多模态线索给出强度/效价估计 | 校准、添加主观判断、在关键转折点人工锚定 |
| 景别/角度/运动分类 | 视觉模型给出标签建议 | 确认、修正、添加"为什么"的判断 |
| 跨镜头模式发现 | 识别视觉/结构上的统计相似性 | 判断哪些相似性是"有意义的母题"而非巧合 |
| 证据检索 | 在用户确立论点后检索相关帧/字幕 | 确认哪些证据真正支撑论点 |

#### 应该全自动化

| 任务 | 原因 |
|---|---|
| 媒体探测（ffprobe） | 纯技术操作，无歧义，无判断 |
| 代理文件生成 | 纯转码，成功/失败无中间态 |
| 缩略图生成 | 确定性操作 |
| 波形缓存 | 确定性操作 |
| 数据库迁移/备份/检查点 | 确定性操作，人不应手动管理 |
| 导出格式化（Markdown/PDF/CSV） | 纯模板操作 |
| 时间码计算和校验 | 无歧义的数学 |
| 编码检测和修复 | 纯技术操作，可失败但不可需要人的判断 |

**关键判断**：当前 spec 的分界基本正确，但在"角色分析"和"声音分析"两个维度上把应该 AI 辅助的任务分配给了纯人工（因为没有对应的数据模型），这是**设计缺陷而非实现缺陷**。

### 4. 现有工具最严重的根本性失败

这里说的不是功能缺失。是**无法通过加功能修复的根本性失败**。

#### 失败 1：时间轴与笔记分离

现有工具（Premiere、DaVinci、Final Cut）把"剪辑/观看"和"分析/笔记"放在两个完全隔离的空间。剪辑在时间轴上，笔记在一个浮窗或外部文档里。这意味着：

- 没有证据绑定：你写"这里的灯光暗示了危险"，但没有东西把这句话和 frame_4832 硬链接。
- 回顾时破裂：6个月后重看笔记，找不到当时看到什么。
- 不可共享为结构化数据：最多导出带时间码的文本，接收方无法导航。

**不可修复性**：这是剪辑软件的架构基因——它们为剪辑（cutting）而建，不是为分析（analyzing）而建。把笔记系统嫁接上去就像给汽车装船帆。Premiere 的 Markers 面板尝试解决这个问题但失败了，因为标记是一维的（一个时间点+一个标签），分析是多维的（结构+情绪+视觉+声音+证据链）。

#### 失败 2：AI 分析黑盒

现有 AI 工具（ChatGPT/Claude 用于影片分析）的失败模式：

- 它在记忆里"看"电影，不真正逐帧处理，所以会编造镜头、混淆时间顺序、遗漏关键画面。
- 它给你一段优美的文字总结，但没有可验证的证据。你无法点击结论跳转到对应帧。
- 它不区分"我看到的"和"我推断的"——混淆证据和判断。

**不可修复性**：这是 LLM 架构的根本限制——它们处理的是语义表征而非时序媒体。你需要一个工具做中介：把媒体拆成帧+字幕+结构，然后让 LLM 在这个结构化上下文中工作。这正是 CineWeave 存在的理由。

#### 失败 3：知识不积累

每次分析都是孤岛。分析完一部片，学到的东西（方法论、模板、对比案例）留在个人脑子里，无法迁移到下一部。后果：

- 影评人写了100篇拉片，但第101篇仍然从零开始。
- 教师每学期重新准备同样的分析框架。
- 个人知识库的价值为零——你无法跨片检索"所有使用负空间结尾的场景"。

**不可修复性**：这是因为现有工具以"影片"为项目单位，而不是以"分析者"为知识库单位。CineWeave 的跨项目检索（V1.1）是正确方向，但前提是数据模型从第一天就能承载这种跨片关系。

#### 失败 4：专业产出物不存在

分析做完后，产出物是什么？一篇 Markdown 文章、一个 PDF 报告、一段视频论文——这些是"展示物"而非"可操作物"。剪辑师想做的是：在分析中标记的镜头能直接作为剪辑参考；教师想做的是：学生可以提交结构化分析作业而非文字报告。

**不可修复性**：这需要统一的语汇和交换格式（类似 FCPXML 但面向分析）。目前没有任何标准。CineWeave 有机会定义这个标准，但前提是先做好，再推动行业采用——这需要3-5年。

### 5. 零基设计：理想中的影视分析工具

不描述功能——描述**体验**。

#### 数据模型的核心思想

数据模型围绕**一个核心概念**组织：**时间轴上任意一点可以附着任意维度的标注，标注之间可以建立关系，关系可以被检索和可视化**。

具体来说：

- 影片是唯一的时间轴。一切存在都以 `timeRange` 为基础。
- 标注（Annotation）是多态实体：可以是镜头边界、结构段落、字幕、情绪点、视觉标签、声音事件、或任意用户定义类型。
- 关系（Relation）是一等公民：`Annotation A → evidence_for → Annotation B`；`Shot A → visual_rhyme_of → Shot C`；`Scene A → contrasts_with → Scene B`。关系类型可由用户扩展。
- 视图（View）是投影：从同一个数据图中裁剪出关注维度——镜头列表是一种投影，结构树是另一种，情绪曲线是第三种，统计图表是第四种。视图之间共享选择状态。
- 知识库（Knowledge Base）聚合所有项目：标签体系、分析模板、用户定义的关系类型、对比结论——这些是用户的**可积累资产**。

#### 交互模型

- **播放即分析**：不要"先看完再做笔记"的分离模式。播放的同时，你可以用键盘快捷键在任何时间轴位置打标签、建关系、标记情绪强度。所有操作都有时间上下文，不打断观看流。
- **证据优先，文字后行**：默认先有视觉/声音证据（帧图、波形片段、字幕行），然后再写文字判断。软件强制这个顺序——你不能创建一个空论点再去找证据，你必须先选中证据再创建论点。
- **渐进式深度**：新用户只需要做两件事——看片，在关键点打标记。然后慢慢发现"我可以把这些标记分组为场景"，再发现"我可以给场景加叙事功能标签"，再发现"AI可以帮我标签化"。深度是逐步揭开的面纱，不是一上来就面对的仪表盘。
- **空间记忆**：利用人的空间记忆。镜头网格不是缩略图列表——它是二维空间布局（按镜头时长决定宽度，按景别换行？按场景分组？按情绪强度着色？），让视觉模式在大尺度上可见。人类擅长在空间中导航，不擅长在树中导航。
- **对比即学习**：默认可以在一个视图中并排两个片段、两个场景、两部电影的相同段落。对比是分析的本质——不是"这一段的属性是什么"，而是"这一段和那一段有什么不同"。

#### 工作流

1. **摄入**：拖入影片。系统探测元数据、生成代理、检测镜头边界、提取字幕。这些都是后台静默发生的。用户在10秒内可以开始播放。
2. **首次观看 = 首次标注**：从头看到尾。在关键点按快捷键打标记。不必现在就分类——标记就像书签，携带时间位置。结束时有100-300个标记点，这就是分析骨架。
3. **结构化**：在标记密集区创建段落。在标记稀疏区补充。给段落命名和叙事功能。这本质上是把时间轴从"连续流"变成"可索引的树"。
4. **深化**：对关键段落展开——逐镜头标注景别、运动、构图、声音设计。这是分析的核心，也是最花时间的部分。AI在这里应该是主动建议者："这段的镜头似乎在越来越近"、"这里的背景音乐在第3个镜头退场"。
5. **关联**：在不同段落/镜头/标记之间建立关系——视觉呼应、主题反复、对照结构。关系图本身就是一个分析成果。
6. **综合**：选择一个模板（教学拉片、导演笔记、影评文章、剪辑参考），系统从数据库中提取相关部分，生成结构化草稿。
7. **发布/导出/复用**：产出是结构化的，不是线性的。可以选择"只导出这支影片的镜头分析"、"导出过去一年所有使用负空间技术的案例"、"生成这堂课的教学材料包"。

### 6. 持久竞争优势（真正的护城河）

#### 什么会随时间增强

| 资产 | 增强机制 | 复制难度 |
|---|---|---|
| **用户的个人分析语料库** | 每分析一部新片，标签体系、关系类型、模板都更丰富。50部片子后的知识库价值远大于工具本身 | **极高**——需要用户花时间积累，不可能 shortcut |
| **社区共享的分析模板** | 如果用户能发布"黑泽明镜头分析模板"、"皮克斯故事节奏模板"并被他人导入使用，模板库就是网络效应 | **高**——Lapian Notes 已经有 MIT 许可，但模板是用户生成内容（UGC），不可自动获得 |
| **时间对齐的细粒度数据** | 越精细的分析，迁移到其他工具越困难。通用 AI 做不到"告诉我《教父》第47镜的景别和叙事功能" | **高**——需要逐片逐镜头的结构化 |
| **跨片对比语料** | "在我分析过的所有王家卫电影中，慢快门的平均持续时长"——这种查询只有积累了数据才能回答 | **极高**——不可购买，只能积累 |

#### 什么不是护城河（错觉）

| 所谓的"护城河" | 为什么是错觉 |
|---|---|
| "我们是 Electron 桌面应用" | Lapian Notes 也是。换技术栈的成本是可管理的 |
| "我们有四种分析视图" | 任何前端团队可以在4周内实现——视图是投影，数据模型才是资产 |
| "我们支持 BYOK" | Twelve Labs API 和其他 AI 视频工具也支持 |
| "我们有暗色 UI" | CSS 变量，一个下午的修改 |
| "本地优先" | 技术选择，非壁垒。Lapian Notes 也是本地优先 |
| "FFmpeg 集成" | 任何桌面视频应用的基础要求，不是差异化 |

#### 真正的护城河公式

```
护城河 = （用户的数据积累深度）×（数据格式的开放性和可迁移性）×（社区模板网络效应）
```

关键悖论：数据积累深意味着迁移成本高——这是锁定而非护城河。真正的护城河是"数据积累了但用户可以带走"——用户留下不是因为切换成本，而是因为**这里有别处没有的、持续创造价值的独特能力**。

CineWeave 的护城河建设路径：
1. 短期（0-6月）：比 Lapian Notes 更好的镜头检测、更强的 AI 证据绑定、更完整的导出
2. 中期（6-18月）：跨项目知识库、分析模板市场、专业交换格式事实标准
3. 长期（18月+）：成为影视教育机构的标准工具、共享语料库、行业认证

#### 为什么通用 LLM 在 2 年内无法替代

不是因为技术能力。是因为**数据结构**：

通用 LLM 即使能完美理解视频，它输出的仍然是文本——它不会生成一个带时间锚的证据链、不会保存镜头级标注数据、不会建立与用户既往分析的关系。LLM 可以写一篇漂亮的影评，但**无法产生可积累、可检索、可对比、可复用的结构化分析资产**。

CineWeave 的价值不在"生成分析"而在"管理分析的产物"——LLM 是内容生成器，CineWeave 是内容管理+关系系统。这两者在未来会协作而非替代。

### 7. 如果这个产品失败，为什么失败？

以下是针对**这个具体产品、这个具体市场**的最可能失败模式：

#### 失败模式 1：用户用了一次就不再用（FATAL）

**症状**：用户导入了一部电影，生成了一些东西，导出了一份报告——然后软件在他硬盘上吃灰。MAU 是 DAU 的 1/30。

**根因**：如果"分析一部电影"是用户一年做1-2次的事情（学生交作业、影评人写稿），那这个产品就没有 DAU 市场。它更接近专业工具（Final Cut Pro，偶尔深度使用）而非日常工具（Notion，每天打开）。

**为什么这很危险**：低频率意味着低付费转化意愿，也意味着用户每次打开都需要重新学习界面，也意味着用户的知识库增长太慢而无法形成锁定效应。

**缓解**：
- 必须让"建立一个镜头知识库"成为一个持续行为而非一次性行为。需要定期导入内容（每周看片+标注？每日影视片段分析？）
- 考虑扩展使用场景：不仅是"分析一部完整电影"，也包括"分析一个片段的视听语言"（短视频级，频率高得多）
- 教学场景可能提供稳定频率——教师每学期重复使用

#### 失败模式 2：Lapian Notes 吃掉市场（HIGH）

**症状**：Lapian Notes 在 MIT 许可下快速迭代，CineWeave 的封闭开发赶不上开源社区的速度。

**根因**：Lapian Notes 已经有可运行产品、有真实用户的反馈、有修复真实 bug 的提交记录。CineWeave 的起点是重写一切——在 Lapian Notes 已经覆盖 70% 功能的情况下，这可能是错误的策略。

**为什么这很危险**：开源 + MIT = 任何人都可以 fork Lapian Notes 并添加"更好的镜头检测"、"SQLite 替代 IndexedDB"、"BYOK 支持"。如果这些改进回流到 Lapian Notes 主线，CineWeave 的价值命题将缩小到"UI 更好看"。

**缓解**：
- 考虑直接在 Lapian Notes 上 fork 和改进，而非平行重建
- 如果坚持独立构建，必须找到 Lapian Notes 一个**无法通过增量改进弥补的架构级差距**作为立足点
- 时间窗口很紧——Lapian Notes 的作者显然在快速迭代

#### 失败模式 3：AI 变得足够好，不需要中间工具（MEDIUM→TRENDING HIGH）

**症状**：2027 年，Claude/GPT 可以直接处理完整电影，给出带时间码的结构化分析，并且比 CineWeave 的输出质量更高。

**根因**：CineWeave 的核心价值之一是"AI 分析的中介层"——拆解影片→打包→让 AI 在结构化上下文中工作→把结果放回时间轴。如果 LLM 可以直接处理视频（原生多模态达到 1M+ token 上下文），这个中介层的价值将大幅降低。

**判断**：这在 12-18 个月内不太可能达到专业级精度（镜头检测准确率 > 95%，不编造不存在的内容，能处理 2 小时长片），但 24-36 个月的窗口内是可能发生的。CineWeave 必须在 18 个月内建立起**独立于 AI 中介之外的价值**——即数据和知识库。

#### 失败模式 4：单人 16 周无法交付可用产品（HIGH）

**症状**：16 周后只有半成品——时间线卡顿、镜头检测在很多场景下不准、AI 导入频繁出错、UI 粗糙。

**根因**：spec 描述的功能范围对于一个独立开发者（即使是 Claude Code 辅助）来说极其庞大。具体来说：

- 镜头检测 + 字幕提取 + 代理生成 = 一个 FFmpeg 重度项目（2-3周调试到各种格式稳定）
- 虚拟化时间线（2小时、1000镜头 > 50fps）= 高级前端性能工程（3-4周）
- AI 双通路 + JSON Schema 修复 + 导入事务 = 大量边缘情况（3-4周）
- Markdown/PDF/CSV/字幕/联系表/FCPXML 导出 = 每种格式都需要调试和测试（3-4周）
- SQLite 迁移 + 检查点 + 恢复 + 路径安全 = 细心活（2-3周）

加上项目库、播放器、四种分析视图、检查器、命令面板、键盘导航、中英文、暗色主题、打包签名——这些每一项看起来都不难，但合在一起就是一个需要3-4人全职6-8个月的项目。

**缓解**：
- 大幅缩减 V1 范围。建议 V1 只做：导入→镜头检测→时间轴→AI 导入→一种分析视图（泳道）→Markdown 导出
- 放弃：结构树、情绪曲线、镜头网格应在 V1.1
- 放弃 FCPXML、PDF 嵌入字体应在 V1.1
- 最重要的：在 Phase 0 结束后（第1周），诚实评估剩余工作量并决定是否缩减范围

#### 失败模式 5：没有 PMF 信号——付费转化率 < 2%（MEDIUM→FATAL）

**症状**：即使免费版有几百用户，愿意付费的几乎为零。

**根因**：
- 目标用户画像太分散（导演+学生+教师+影评人+游戏过场制作人），没有一个人群足够大或足够痛
- 核心价值主张"AI辅助分析+结构化"对普通影迷太专业，对专业人士可能太浅
- 免费版已经覆盖了最常用路径（手动 AI 包+基础时间线+Markdown 导出），Pro 增加的是"高级功能"而非"解决免费版中的痛苦"

**为什么这很危险**：如果免费版对 90% 的用户足够，你就是一个免费工具——没有商业可持续性。

**缓解**：
- 免费版必须有使用限制（项目数量、导出水印、单项目镜头数上限），驱动升级
- 找到付费意愿最明确的人群，专门为他们打造——可能是影视专业教师（需要教学模板、评分功能）或职业影评人（需要 PDF 导出、品牌水印、批量处理）
- 在 Beta 阶段就测试付费意愿——不要等 V1 发布

### 8. 我们做了哪些可能是错误的假设？

逐一验证每个隐含假设：

| 假设 | 挑战 |
|---|---|
| "影视分析是一个足够大的独立需求，值得单独一个工具" | 可能不是。对多数从业者而言，影视分析是"为某个目的服务的手段"——为写论文、为教学、为剪片、为写剧本。如果分析不是为了产出，那它只是一个昂贵的爱好。你需要验证：**有多少人会把"分析一部电影"作为一个独立的、离散的、有开始和结束的活动？** |
| "创作者想要本地优先，不上传媒体" | 可能部分正确但商业上危险。本地优先意味着无云收入、无协作、无跨设备同步——这些不是 V1 的问题，但会限制天花板。而且，如果"协作审片"是更痛的刚需（导演+剪辑师+制片人的工作流），本地优先直接排除了这个市场。 |
| "影片不上传"这个隐私价值够大" | 对于未发布素材（正在拍摄/剪辑中的影片），是的。对于已上映的电影（影评人、学生分析的素材），用户根本不关心隐私——他们关心效率。如果云端分析更快更好，他们会上传。你的隐私卖点只对 20% 的用户有效。 |
| "手动 AI ZIP 包是一个可行的用户体验" | 对于技术用户（开发者、懂脚本的学生）可行。对于普通学生或人文学科的教师，这是摩擦。BYOK 配置、API Key 管理、模型选择——每一步都是漏斗收缩点。如果 70% 的潜在用户在"配置 AI"步骤放弃，你有市场问题。 |
| "Electron 能承载媒体重度应用" | 在其他领域有成功案例（VS Code、Figma、Discord），但视频播放+Canvas 渲染时间线+大量图像缩略图是不同的问题。内存管理（每部电影加载后占多少 RAM？）、GPU 加速在 Electron 中的可用性、跨平台解码器的实际能力——这些是工程风险，需要快速原型验证。 |
| "12-16 周足够" | 如失败模式 4 所述——几乎肯定不够。建议：**不要基于 roadmap 的乐观版本做商业决策**。给最悲观估算翻倍，然后看是否仍然可行。 |
| "Free + Pro 商业模式可行" | 免费桌面软件在 Windows/macOS 上的变现记录很差。用户期望桌面软件是"一次付费"的（即使现在已不再主流），订阅模式在桌面端的接受度低于 SaaS。你需要验证：**目标用户愿意为桌面软件付费的金额上限是多少？他们上次为哪个桌面工具付费？** |
| "Claude Code 能有效协助构建" | Claude Code 擅长的领域（代码片段、自动化、模板）和困难领域（性能调优、时间线虚拟化、FFmpeg 边缘情况调试）有本质差距。最困难的工程决策需要人的判断。不能把 Claude Code 当成第二个开发者——它更像是熟练但无知觉的初级助手。 |

## 第二部分：对抗性审查

### 市场可行性

#### 审查点 1："详细影视拆解"是独立产品还是功能？

**裁定：可能是功能，需要证明它是产品。**

证据：
- 所有主流的非线性编辑软件（Premiere, DaVinci, Final Cut）都有标记/笔记面板——虽然粗糙，但说明这个需求已经被识别为 NLE 的一部分
- Notion/Obsidian/Logseq 被大量用于影视笔记——说明人需要结构化分析，但现有专业工具不够好
- Lapian Notes 的存在证明至少有一些人把这当作独立需求

**反方证据**：
- Lapian Notes 发布后没有公开 Issue/PR（虽然"首发后快速迭代"是事实，但没有社区参与说明用户群极小）
- StoryFit 在 2025 年中因"低行业采用率"失败——这是一个有融资的 AI 剧本分析公司，它失败了不是因为没有 AI，是因为没有人在按它设想的方式工作

**缓解**：CineWeave 不是 StoryFit（面向制片公司的剧本评估），这是面向创作者的分析工具。但需要诚实地问：**"影视拆解工作台"的 TAM 是多少？** 做一个估算：

- 全球电影学院/影视专业学生：约 50-80 万（中国约 15-20 万）
- 活跃影评人/内容创作者（以此为职业或半职业）：约 5-10 万
- 专业导演/剪辑师/编剧：约 20-30 万
- 电影爱好者（愿意深度分析）：约 10-20 万
- 培训机构和教师：约 1-2 万

**乐观 TAM：~100 万潜在用户。实际可触及市场（SAM）可能只有 5-10 万**（需要他们知道这个工具、愿意下载试用、愿意改变现有工作流）。

**严重性：MEDIUM → 如果 TAM 验证后 < 50万，则升至 HIGH**

#### 审查点 2：有人会为此付费吗？

**裁定：会，但人数很少，客单价低。**

对标产品分析：
- Final Draft（剧本软件）：$249.99 买断。年用户估计 30-50 万
- Frame.io（协作审片）：月费 $15-$25/用户，有清晰的企业 Tier
- Notion：免费版足够，Plus $10/月
- Lapian Notes：免费 MIT——零收入

这些人为什么付费？
- Final Draft：行业标准，不付费无法参与专业剧本流通
- Frame.io：解决了协作审片的真实物流痛点
- Notion：日常高频使用的生产力工具

CineWeave 的付费理由是什么？"更多导出格式"、"BYOK"、"跨项目搜索"——这些是 nice-to-have，不是 painkiller。没有清晰的"不付费就做不了"的场景。

**严重性：HIGH —— 需要在 Beta 阶段验证付费意愿**

**推荐**：
- 把 BYOK 放进 Free 版（降低上手摩擦），Pro 卖"知识库规模"（项目数、存储、对比功能）
- 寻找机构付费场景（学校 site license、培训机构批量）
- 考虑"对个人免费、对机构收费"的 Red Hat 模式

### 竞争威胁评级

| 竞争者 | 威胁级别 | 理由 |
|---|---|---|
| **Lapian Notes 成熟** | **FATAL** | MIT 许可、已有可运行产品、相同的技术栈和产品思路。最危险的竞争——它是开源的，可以 fork 并添加 CineWeave 的差异化功能，而 CineWeave 不能 fork 并挪用它的用户群。CineWeave 只有 12-16 周的窗口在 Lapian Notes 补上自身短板之前建立足够的差异化。 |
| **ChatGPT/Claude 多模态增强** | **HIGH → 12个月后升至 FATAL** | 当前 LLM 无法准确处理 2 小时长视频的逐帧分析。但 12-18 个月内，原生多模态+超长上下文将大幅缩小差距。CineWeave 必须在 12 个月内建立独立价值。 |
| **Twelve Labs API + 薄封装** | **MEDIUM** | Twelve Labs 的视频理解 API 正在快速改进。一个技术创业者用 4 周就能做一个 Web 版"AI 电影分析器"调用此 API。但它不做离线、不做隐私保护、不做本地数据库——这些是差异化。不过对于不在意隐私的用户（多数影评人），Web 版可能更方便。 |
| **DaVinci Resolve AI 增强** | **MEDIUM** | DaVinci 拥有完整的剪辑-调色-音频-交付管线，如果他们在分析工作台方向上投入，天然优势巨大。但 Blackmagic 的战略重心不在此——他们在做硬件（摄影机）+ 编辑软件生态。分析工作台对他们来说是小众功能，优先级低。 |
| **Adobe 在 Premiere 中加入分析** | **MEDIUM** | Adobe 有 Frame.io 用于审查，有 Sensei AI，有 Premiere 的用户基础。他们在片场到发布到协作的完整链条上有位置。但 Adobe 产品有订阅费用、体量大、面向专业流水线——对个人学生和影评人是过度方案。 |
| **中国竞品克隆** | **LOW→MEDIUM** | 如果 CineWeave 在中国市场有 traction，6个月内会出现 3-5 个克隆品。但有 2 个缓冲：(1) 它们通常质量差；(2) 它们会忽略海外市场。如果 CineWeave 先建立用户数据积累（护城河），后发者的功能克隆无法弥补数据差距。 |

**Lapian Notes 专题分析**：

Lapian Notes 的致命弱点（CineWeave 可以进行差异化攻击的方面）：
1. IndexedDB 是玩具级存储——最大容量、事务可靠性、迁移能力都差
2. 固定 1 秒抽帧无法表达镜头——这是 Lapian Notes 的核心架构缺陷
3. Vite dev server 做转码后端——生产环境无解，需要 Electron 化
4. 没有真正的 AI 返回校验和修复——JSON 可能被模型破坏
5. 没有检查点/版本管理
6. 没有专业导出格式

**CineWeave 对 Lapian Notes 关键几周内的动作**：
- Lapian Notes 的作者正在快速迭代（从提交记录推断）
- Lapian Notes 的修复是"用户驱动的紧急性修复"而非"架构前瞻性改进"
- Lapian Notes 没有商业模式——作者可能不会长期维护
- 但如果 CineWeave 没有在 Lapian Notes 补上短板之前发布可用版本，CineWeave 就是 Lapian Notes 的一个更复杂但没有发布的开源替代品

### 技术怀疑论

| 审查点 | 结论 | 严重性 |
|---|---|---|
| **Electron 适合媒体重度应用吗？** | 部分适合。VS Code 证明了 Electron 可以是性能优异的。但视频播放 + Canvas 渲染 + 大量缩略图的内存模型与代码编辑完全不同。关键风险：渲染路径（硬件解码→Canvas vs Video 元素）、内存占用（1000+ 缩略图）、跨平台解码器能力差异。 | **HIGH — 需要 W1 技术验证** |
| **FFmpeg 在各平台和格式的可靠性？** | FFmpeg 本身成熟可靠，但跨平台的集成痛点：(1) Windows 缺少原生解码器，需要打包解码库；(2) 不同版本的 FFmpeg 行为差异；(3) 视频格式的尾部情况（损坏的 MKV、不标准的 MOV、手机拍摄的变帧率视频）非常多。Lapian Notes 已经遇到的这些问题正是证明。 | **MEDIUM — 已知可管理但投入时间被低估** |
| **SQLite 能处理 2 小时电影+1000+ 镜头？** | 完全够。SQLite 的单文件读写比 IndexedDB 优秀得多。1000 镜头 = 千级记录，对 SQLite 来说不构成挑战。真正的问题是：虚拟化时间线渲染（不是 DB 查询，是 Canvas 渲染 1000 个镜头的轨道），以及全文检索在中文环境的表现。 | **LOW — SQLite 不是瓶颈，前端渲染才是** |
| **镜头检测在动画、暗场景、快速动作中的准确性？** | FFmpeg 的 `select='gt(scene,0.4)'` 在这些场景中准确率急剧下降。动画：纯色块+线条变化，场景检测易漏。快速动作：连续高变化率，场景检测易误触发。暗场景：视觉差异小，几乎无法检测。真实的镜头检测准确率在没有 AI 辅助的情况下，对好莱坞电影约 85-90%，对动画约 60-70%，对黑暗场景约 40-50%。 | **HIGH — 镜头检测是数据分析的基石，基石不准，上层全偏。需要投入 PySceneDetect 或其他专用方案** |
| **桌面应用能否处理所有媒体格式？** | 不能。这是硬件层面的事实。Windows 不支持 ProRes 编码（需要额外许可），HEVC 解码器在 Windows 上需要商店安装的扩展或额外 DLL，高质量 10-bit 素材的播放性能。如果能播放但不能解码，就需要代理文件——代理文件的生成时间可能让人无法接受（2 小时电影在普通笔记本上可能需要 30-60 分钟）。 | **MEDIUM — 创建代理文件时长的预期管理是关键 UX 问题** |

### UX 怀疑论

| 审查点 | 结论 | 严重性 |
|---|---|---|
| **四个视图 + 时间线 + 检查器 — 新用户是否被淹没？** | **确定无疑**。Lapian Notes 的三个面板（左导航、中央时间线、右侧检查器）已经信息密集。CineWeave 增加了更多视图、更复杂的 AI 集成、更多的选项。对于影视专业的学生（核心用户之一），如果他们打开软件 5 分钟内无法看到价值，他们会卸载。 | **FATAL — 需要一个极度简化的首次使用模式，隐藏 70% 的功能直到需要时** |
| **桌面应用 vs Web — 哪个是正确选择？** | 对于"保护未发布素材隐私"和"使用本地 GPU 进行转码"来说，桌面应用正确。但对于：(1) 学生（在教室/图书馆/家里切换设备），(2) 协作审片，(3) 快速试用（下载 500MB 安装包 vs 打开网址），Web 有明显优势。**正确的答案可能是：桌面做核心分析，Web 做只读查看和轻量编辑**——但这是 V2 的范围。 | **MEDIUM — V1 走桌面路线可行，但需为未来 Web 扩展留空间** |
| **专业用户会从现有工具切换吗？** | 对于导演和剪辑师（已经在使用 Premiere/Resolve）：不会，除非 CineWeave 与他们的工作流有双向集成（能导出到 NLE，能从 NLE 接收标记）。对于学生（还没有固定工具）：会，如果老师和课程推荐。对于影评人：可能，如果导出的报告质量明显高于 Markdown。 | **MEDIUM — 把"获得首批用户"的希望放在学生的自然传播上可能是合理的策略** |
| **手动 AI 打包方式是否摩擦太大？** | 对于技术用户（懂 ZIP/AI API/JSON）：可接受但麻烦。对于不技术的用户（人文学科学生、老导演）：可能是一道不可逾越的墙。你的核心用户中至少有 30-40% 无法或不愿完成"打包→上传→下载→导入"的流程。 | **HIGH — BYOK 应该是默认路径，手动 ZIP 作为后备。不应该在默认教程中把手动 ZIP 作为首选** |

### 商业模式

| 审查点 | 结论 | 严重性 |
|---|---|---|
| **Free + Pro 转化率** | 桌面软件（非 SaaS）在 Free→Pro 上的历史转化率通常是 2-5%。假设 10 万安装，Pro 用户 2000-5000 人。如果 Pro 定价 $5/月，年收入 $12-30 万。这对于一个独立开发者来说可以维生，但无法支撑团队扩张、服务器成本、营销。需要考虑：这是生活方式业务还是 VC 级别的业务？ | **MEDIUM — 先验证能否养活一个人** |
| **AI 打包方式：聪明区分还是 UX 摩擦？** | 这是一个**好的长期保护**（避免 AI 服务成本吞噬利润），但不应成为用户的唯一 AI 入口。建议：**BYOK 是默认路径，内嵌免费额度（赠送前 3 次直连分析），手动打包是高级用户可以随时使用的逃生舱** | **MEDIUM — 调整后可从弱点变成特色** |
| **AI 一个 prompt 就能做到的那一天？** | 当 AI 能直接分析完整电影时：(1) CineWeave 的"AI 中介"价值归零；(2) 但"数据管理+知识库+跨片检索+专业导出"价值不变。关键是判断这两个价值在"AI 足够好"的那一天哪个更大——如果 80% 的用户价值来自 AI 中介，那这是一个烧钱的倒计时。如果 60%+ 来自数据和知识管理，那可以轻松面对这个未来。 | **HIGH — 核心定位需要更偏向"知识管理"而非"AI 分析"** |

### 具体 SPEC 差距

这些都是 spec 中**没有讨论**或**讨论不充分**的领域：

| 差距 | 影响 | 严重性 |
|---|---|---|
| **没有协作审片/反馈功能** | 导演、剪辑师、制片人之间的工作流天然需要协作。CineWeave 是单人工具——这意味着排除了"专业工作室"这个市场。如果目标用户是学生/个人，这可行。但在 spec 中写"暂缓企业版"的同时，没有说明**V1 用户如何分享分析结果给其他人查看**——没有"只读分享"功能。 | **HIGH — 至少需要"导出为可分享的只读包"** |
| **没有声音分析数据模型** | spec 提到"波形缓存"但没有声音事件（音乐入场、音效、动态范围、对白/环境声分离、Loudness 曲线）。导演和剪辑师非常关心声音，声音至少占分析价值的 30-50%。没有声音模型，"音频峰值缓存"就是只读不分析的摆设。 | **FATAL — 数据模型需要现在而不是 V2 加入声音事件实体** |
| **没有色彩分析** | 色彩分级是影视分析的核心维度之一。白平衡变化、色调映射、LUT 风格、亮度曲线——这些都是可以被分析和标注的元素。spec 中镜头字段有"色彩"但没有色彩分析对应的数据模型。 | **MEDIUM — 至少需要可标注的色彩标签和跨镜头的色彩变化检测** |
| **没有 TV 剧集/多集支持** | 如果只分析单部电影，就排除了以下市场：电视剧（全球最大的影视生产形式）、系列电影分析、短剧、广告系列。单集分析不等于系列分析——系列分析需要跨集的角色弧线追踪、结构模板比较、收视/评价与叙事分析的联系。 | **MEDIUM — V1 可以不支持，但数据模型应该用 `project_group` 为未来多集留位置** |
| **没有考虑非电影内容的确切需求** | spec 说目标包括"广告和游戏过场的导演"，但广告（15-60秒）和电影（90-180分钟）的分析工作流完全不同。广告需要帧级精度（每秒的决策都是深思熟虑的），分析重点是商业目标、品牌消息、CTA 框架——这不是电影分析模型能覆盖的。要么明确限缩到"电影+剧集"，要么诚实评估其他内容类型需要独立的数据模型。 | **LOW — 建议限缩到电影+短片，其他后续** |

### 执行风险

| 风险 | 分析 | 严重性 |
|---|---|---|
| **12-16 周完成所有功能** | 如前述失败模式 4——**不可能**。即使 Claude Code 能写 60% 的代码，调试、集成测试、跨平台测试、性能调优、12 种对抗性场景的修复——这些都需要人的时间。更诚实的估算是：**内部 MVP（仅核心路径）6-8 周，可用 V1（当前 spec 的功能）需要 20-24 周**。 | **FATAL — 必须缩减 V1 范围** |
| **FFmpeg 打包到桌面应用** | 技术上可行（Electron Forge/electron-builder 支持二进制打包），但有三个隐形成本：(1) 安装包大小增加 80-120MB；(2) Windows/macOS 代码签名对二进制的要求；(3) FFmpeg 版本升级的安全响应。如果 FFmpeg 发现严重漏洞，你需要推送整个应用更新，不能只更新 FFmpeg。 | **MEDIUM — 需要封装 FFmpeg 调用层，支持系统安装的 FFmpeg 作为备选** |
| **跨平台媒体处理** | Windows 和 macOS 在以下方面有根本差异：(1) macOS 有 VideoToolbox 硬件编解码，Windows 需要 DXVA2（不同 API）；(2) Windows 缺少 ProRes 编码器；(3) macOS 可以通过 AVFoundation 读一些 FFmpeg 不支持的格式。这意味着你可能需要两个平台的编解码路径——或者统一用 FFmpeg 软件编解码（慢但可维护）。 | **HIGH — 在 W1 需要在两个平台上测试 10 种常见格式，确认软解码性能是否可接受** |

## 总结：致命问题清单（按优先级排序）

### 现在就要回答的问题（否则不应写一行代码）

1. **[FATAL] 为什么不 fork Lapian Notes 然后改进？** 这是最重要的战略问题。Lapian Notes 是 MIT 许可的、已经可运行的产品。你是在重复 60-70% 的工作来重建它已经有的功能。如果你 fork Lapian Notes 并做以下改进：SQLite 替代 IndexedDB、改进的镜头检测、AI Schema 校验、专业导出、暗色主题——你在 4-6 周内就能有一个 V1，而不是 16 周。**说服力不足的回答包括**："代码架构不够好"、"我们用不同的技术选型"、"我们想从零开始控制所有"。**唯一合理的回答**："Lapian Notes 的核心架构有一个不可修复的设计缺陷，使得 [具体说明] 无法在它的基础上实现"。

2. **[FATAL] 12-16 周的估算是基于什么？** 需要逐周、逐功能分解并估计，然后与有 Electron+FFmpeg 开发经验的人交叉验证。如果答案是"基于 spec 的 phase 划分，加上每个 phase 的粗略天数估算"，那么这个估算没有经过现实检验。

3. **[FATAL] V1 的最小可行范围是什么？** 当前 spec 中的 V1 有 12 个 FR、4 个分析视图、6 种导出格式、AI 双通路——这不是 V1，这是 V2。真正的最小 V1 可能是：导入→镜头检测→一个视图→AI 导入→一种导出格式。把这个做稳，然后再加其他。

4. **[HIGH] TAM 验证的证据是什么？** 不能基于"我认为有 N 万潜在用户"。需要：与 10+ 目标用户的深度访谈（不是调查问卷）、Lapian Notes 的 GitHub stars/fork 增长趋势分析、影视专业课程大纲中"拉片分析"的占比和频率数据。

5. **[HIGH] 谁会是第一个付费用户？愿意付多少钱？** 在写代码之前找到 5 个明确表示"我会为这个工具付费"的人——至少要有他们的邮箱、真实身份和具体使用场景。没有这个，你是在为假想的用户构建产品。

### 建议的立即行动

1. **如果决定独立构建**：缩减 V1 范围为"核心路径 MVP"（导入、镜头检测、时间轴、AI 导入、Markdown 导出、一个分析视图），目标 8 周。将其余所有功能移到 V1.1 或后续。

2. **评估 fork Lapian Notes 的可行性**：用一个下午仔细阅读 Lapian Notes 的核心架构文件（`App.tsx`、`FrameTimeline.tsx`、`aiImport.ts`），确定是否可以在其基础上增量改进而不是重写。

3. **技术原型验证（第 1 周完成）**：
   - FFmpeg 镜头检测在 5 种不同类型片段上的准确率（好莱坞、独立电影、动画、纪录片、MV）
   - Electron + Canvas 渲染 1000 个虚拟化镜头的 FPS 数据
   - 1 小时电影的代理文件生成时间
   - Windows/macOS 双平台 10 种媒体格式的 ffprobe 探测成功率

4. **用户验证（第 1-2 周完成）**：
   - 与 10 个影视学生/影评人做深度访谈，展示 Lapian Notes 做竞品参照，询问"你会为什么功能付费？"
   - 制作一个 Figma 原型替代当前 spec 的 UI 描述，进行可用性测试

5. **商业模式验证（Beta 前完成）**：
   - 设置 Landing Page + 等待列表
   - 设置 Stripe 支付页面，测试用户是否真的会点击"购买"（即使不实际收费）
   - Free/Pro 功能分界需要和潜在用户一起确定，而不是 spec 的内部推测

### 最后的话

这是一份非常好的 spec——结构清晰、技术选型扎实、对抗性场景考虑周到。它也是一个不罕见的现象：一个经过深思熟虑的 spec 试图在 16 周内完成一个 32 周的项目，目标是 100 万用户市场中可能存在的 5 万用户，与一个 MIT 许可的、已经可运行的开源项目竞争。

这不一定意味着应该放弃。但意味着在写第一行代码之前，面临三个"不"的选择：

- 不大幅缩减 V1 范围 → 不现实
- 不平移对 Lapian Notes 的策略 → 浪费 60% 的工作量
- 不验证 TAM 和付费意愿 → 在赌一个可能根本不存在的市场

如果这三个问题有诚实的、有证据支撑的回答，这个产品有成立的可能性。CineWeave 在知识管理和结构化分析资产上的远见是正确的——问题不是方向，而是到达方向之前需要先活下来。
